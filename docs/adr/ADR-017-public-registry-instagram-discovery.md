# ADR-017: Public registry Instagram discovery (FHRS + event directories)

- **Status:** Accepted and implemented for operator-triggered testing
- **Context:** ADR-016 resolves an Instagram professional profile only when the operator already has its URL; Meta's API has no keyword/location search. That leaves the highest-value segment — home bakers and home-based event/wedding planners with no Google Places listing and no website — structurally unreachable, since Instagram is often their only public presence.
- **Decision:** Add one operator-facing discovery source, "Public registries & directories," selected once per campaign. Internally it routes to whichever registry sources match the campaign's `segment`/`keywords`/`product_categories` against a static, engineering-owned matcher table (`_REGISTRY_MATCHERS` in `providers.py`) — the operator never picks FHRS vs. event directories individually, and a campaign matching no registry completes with a clear warning rather than doing nothing silently. Two registries ship in this increment, both used only to *generate candidate handles/URLs*; every resulting identity is still resolved exclusively through Meta's official Business Discovery lookup (`MetaInstagramProvider.lookup_profile`) — neither registry ever touches Instagram directly, and the scraping/browser-automation/unofficial-API prohibitions from ADR-016 are unchanged:
  - **FHRS** — the UK Food Standards Agency's free, keyless `Establishments` API (`x-api-version: 2`, no registration required), queried by the campaign's `primary_location`. UK law requires anyone selling food, even from a home kitchen, to register with their local authority, so this reaches home bakers/caterers with no other online footprint. Business names are turned into a small set of plausible Instagram handles (`generate_handle_guesses`) and each is verified through Meta; only an unambiguous single match is promoted.
  - **Event directories** — there is no legal registration requirement pulling unincorporated event/wedding planners into a government database, so the closest public equivalent is directory pages already published by councils and marketplaces. A small, manually verified catalog of such pages (currently: the Suffolk Registration Service supplier directory) is fetched with the existing `SafeWebsiteEnricher` (robots.txt-respecting, SSRF-hardened, byte/time capped), followed one hop into each listed supplier's own page, and any Instagram links found are verified through the same Meta lookup.
- **Credentials:** None new. FHRS is free and keyless. Event-directory pages are public HTTP fetches through the same enrichment path already used for a lead's own website. Both still require an existing Meta connection, since every candidate is verified through it.
- **Limitations:** Handle-guessing is heuristic — a business name may not match its real handle, and Meta's API cannot distinguish "this guess doesn't exist" from "rate-limited," so guesses are bounded per business and per run, and a run stops early with an explicit warning after repeated unresolved lookups rather than reporting a false "no matches." FHRS stops at a business's first successful guess rather than trying every guess and checking for ambiguity — trying more guesses after already finding a match spends more of the lookup budget without reducing the risk of a coincidental false match, since that risk lives in the individual guess, not in how many more are tried afterwards; a wrong guess that happens to resolve to someone else's real account is a residual risk inherent to any name-based guessing, not something either strategy eliminates. The event-directory catalog is deliberately small and grows only by manually verifying a new source's page structure, not by guessing a commercial site's URL pattern — several considered directories (Hitched, Bridebook) were not added because their listing-page structure could not be verified from this environment. A campaign matching no registry is surfaced as a warning rather than treated as an error.
- **Consequences:** Operators tick one checkbox per campaign; which registries actually run, and how many are added over time, is a code change, not per-campaign configuration. Provider failures from one matched registry (e.g. FHRS being unreachable) do not block a second matched registry's results in the same run; only a systemic Meta connection failure aborts the whole run.
- **Rollback:** Remove `public_registries` from a campaign's discovery sources, or disconnect Meta in Settings (both registries require it and no-op without it). Canonical leads and their source evidence remain available locally.
